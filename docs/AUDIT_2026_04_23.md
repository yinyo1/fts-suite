### 3.1 Bug timezone UTC (`toISOString().split('T')[0]`) — 🟡 ALTO

Patrón que retorna la fecha en UTC en lugar de la fecha local del navegador. Monterrey (CST UTC-6) produce fecha adelantada 1 día después de las 18:00 local. **9 ocurrencias**:

- `operaciones/config/index.html:902`
- `operaciones/dashboard/js/dashboard.js:144` — dashboard muestra "sin técnicos" tras 18:00 CST
- `operaciones/incidencias/empleado/index.html:233, 357`
- `operaciones/incidencias/supervisor/index.html:205, 368, 441`
- `shared/auth-suite.js:84` — validación de `vigencia` de usuarios con fecha UTC

**Nota**: PR #4 ya arregló las 2 ocurrencias críticas del kiosk (`kiosk.js:1376, 1518`) con el helper `fechaLocalISO()` — `operaciones/kiosk/js/kiosk.js:38–46` documenta el problema y el workaround. Las 9 restantes aún usan el patrón buggy. **Solución pendiente**: extraer `fechaLocalISO()` a `shared/date-utils.js` y reemplazar los 9 call sites.

### 3.2 Comparaciones `==` en contextos críticos — ✅ OK

**6 ocurrencias totales**, todas son `== null` (captura `null` + `undefined` en un solo check) — patrón idiomático seguro. **No hay bugs introducidos por coerción de tipos.** No requiere acción.

### 3.3 GitHub PAT en localStorage (`ops_github_token`) — 🟡 ALTO

**19 referencias** totales (lectura + escritura), dual-write a `ops_github_token` + `key_github_token` sin cifrado.

**Lecturas críticas**:
- `operaciones/config/index.html:865, 1099`
- `operaciones/incidencias/js/incidencias.js:145, 181`

**Paso 2 — `syncUsersToGitHub()`** replica el patrón `syncGeoGitHub()`:
```js
async function syncUsersToGitHub(){
  const token = lsGet('ops_github_token');
  if(!token) throw new Error('Sin GitHub PAT (configúralo en Sincronización)');

  const API_URL = 'https://api.github.com/repos/yinyo1/fts-suite/contents/shared/users-suite.json';
  const headers = {
    'Authorization': 'token ' + token,
    'Accept':        'application/vnd.github.v3+json'
  };

  async function getRemote(){
    const res = await fetch(API_URL + '?ref=main&t=' + Date.now(), { headers: headers });
    if(!res.ok) throw new Error('GET users: ' + res.status);   // ← línea 2167
    const data = await res.json();
    const json = JSON.parse(decodeURIComponent(escape(atob(data.content.replace(/\n/g,'')))));
    return { sha: data.sha, json: json };
  }

  let remote = await getRemote();           // 1) GET SHA + snapshot completo
  remote.json.users = USERS_STATE;          // 2) sobreescribe solo users[]

  let resPut = await fetch(API_URL, {       // 3) PUT con SHA
    method: 'PUT',
    headers: Object.assign({}, headers, { 'Content-Type':'application/json' }),
    body:    buildBody(remote.sha, remote.json, false)
  });

  if(resPut.status === 409){                // 4) 409 retry con SHA fresco
    remote = await getRemote();
    remote.json.users = USERS_STATE;
    resPut = await fetch(API_URL, { ..., body: buildBody(remote.sha, remote.json, true) });
  }

  if(!resPut.ok){
    let err = {};
    try { err = await resPut.json(); } catch(e){}
    throw new Error('PUT ' + resPut.status + (err.message ? ' — ' + err.message : ''));
  }
}
```

**Paso 3 — UX feedback**: el modal muestra estados secuenciales (`⏳ Validando…` → `⏳ Guardando en GitHub…` → `✅ Guardado`) y cierra con `setTimeout(closeUserModal, 700)` tras éxito. Errores mantienen el modal abierto con texto rojo para retry manual.

Esperado: `HTTP/2 200` + `content-type: application/json`. Si retorna 404, el problema reportado anteriormente sigue vigente y los modelos de face deben moverse a otra ubicación (mirror propio en GitHub raw, p.ej.).

### 7.4 Dependencias críticas y fallbacks

Para cada dependencia user-facing, evaluar si hay fallback:

| Feature | Dependencia | Fallback presente | Comportamiento si falla |
|---|---|---|---|
| Verificación facial en kiosk | `face-api.js` 0.22.2 + weights | ❌ Ninguno (soft-fail en `face.js:39–44`) | El kiosk continúa SIN verificación facial, mostrando warning. Funcional pero sin segundo factor biométrico. |

| Anthropic API (análisis estructural) | `api.anthropic.com` | ❌ Sin retry, sin fallback | Análisis falla; el módulo es inutilizable hasta que vuelva. |

### 7.5 Riesgos identificados

| Riesgo | Severidad | Detalle |
|---|---|---|
| Anthropic key expuesta en cliente | 🔴 CRÍTICO | Ver SECCIÓN 3.5 — cualquiera puede extraer y usar. Costo + reconnaissance. **Acción inmediata: revocar + proxy.** |
| GitHub PAT expuesto en localStorage | 🟡 ALTO | 19 referencias en código; vulnerable a XSS local. **Acción: proxy n8n para users (pendiente #1).** |
| CDN third-party sin SRI (Subresource Integrity) | 🟡 ALTO | Si jsdelivr / cdnjs son comprometidos, ejecutan código arbitrario en todos los kioscos. Frontend no usa atributos `integrity=`. |
| Face-api CDN puede romperse (404 reportado previamente) | ⚠️ MEDIO | Verificar TODO de 7.3. Si está roto, soft-fail evita crash pero degrada seguridad del kiosk. |
| Sin retry en Anthropic / Odoo / GitHub API | ⚠️ MEDIO | Una falla transitoria de red rompe la operación; usuario debe retry manual. n8n sí tiene retry interno (3 intentos exponential backoff en algunos endpoints). |
| Versiones pineadas sin proceso de update | ⚠️ MEDIO | Sin dependabot ni equivalente. Si jsPDF 2.5.1 tiene una vulnerabilidad, no hay alerta automática. |

### 7.6 Inventario de servicios externos vs control del repo

**Fuera del control del repo** (third-party crítico):
- jsdelivr, cdnjs, fonts.googleapis (CDNs)
- Anthropic (modelo IA)
- GitHub Pages / GitHub API (hosting + persistencia)
- n8n on Railway (workflows)
- Odoo on serviciosfts.odoo.com (source of truth de attendance)

**Dependencias internas** (controladas por FTS):
- `shared/users-suite.json` (auth source)
- `shared/public-config.json` (geolocations, n8n_url)
- `shared/incidencias.json` (log append-only generado por n8n bot)

**Implicación**: una caída de jsdelivr o Railway tumba el kiosk. Considerar:
- Mirror local de face-api weights + librerías PDF/Excel en `shared/vendor/` para reducir dependencia externa.
- SRI hashes en los `<script src=...>` para mitigar CDN poisoning.
